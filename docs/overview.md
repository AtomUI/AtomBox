# AtomBox 文档

> 文档状态：第一版发布基线冻结
>
> 冻结时间：2026-06-22
>
> 冻结范围：当前文档定义的文档目录职责、工程治理要求和第一版发布基线文档入口
>
> 变更规则：实现阶段不得随意修改本文件；如需调整本文档定义的设计边界，必须先说明原因，再同步更新相关设计文档。

本目录维护 AtomBox 软件的架构设计与工程实现说明、包括每个模块的技术实现细节约定和使用指南。

## 全局强约束

AtomBox 软件采用文档先行研发规则。

所有模块必须有完善的文档后，才能开始写代码。任何改动都必须先完成文档对齐，并确认文档没有问题后，才能进入实现。

该规则是项目级工程治理要求，不是建议。当前 v0.1 发布前文档同步以已冻结代码为事实源，只允许修正文档落后内容，不修改代码。

## 文档说明

| 目录 | 职责 |
|---|---|
| `architecture/` | 框架顶层架构、包边界、编程范式、生命周期和依赖策略。 |
| `modules/` | 各模块的设计文档。 |
| `ui/` | Desktop 低保真 UI 线框图、页面布局和交互草图。 |
| `implementation/` | 实现路线入口、Phase 阶段归档、测试矩阵和发布前验收说明。 |
| `test-reports/` | 阶段性测试结果和已知问题记录。 |

## 生命周期文档

生命周期设计统一记录在 `Docs/architecture/lifecycle.md`。

任何涉及业务对象存活时间、运行时服务注册方式、模块启动停止顺序、后台服务边界的实现，都必须先阅读生命周期文档。

DI 注册方式只能作为生命周期设计的落地结果，不能反过来决定对象或模块应该活多久。
