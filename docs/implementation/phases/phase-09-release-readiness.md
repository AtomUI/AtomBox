# Phase 9 Release Readiness 验收规划

> 文档状态：阶段性冻结
>
> 冻结时间：2026-06-15
>
> 冻结范围：Phase 9 产品可试用性收口的执行切片、验收清单、能力矩阵和结束判定
>
> 变更规则：实现阶段不得随意扩展 Phase 9 范围；如需调整，必须先同步更新 `roadmap.md` 和相关 architecture、modules、ui 文档。

本文档是 Phase 9 的执行验收清单。`roadmap.md` 定义阶段边界，本文档定义 Phase 9 做完时要逐项确认什么。

## 1. Phase 9 总目标

Phase 9 的目标是把 Phase 8 后已经贯通的工程 MVP 收敛成可试用 MVP。

可试用 MVP 的最低标准：

- 用户危险操作前有确认。
- 用户失败后能看到摘要和脱敏详情。
- 用户知道哪些任务成功、失败、取消或中断。
- 应用启动失败时不进入半坏的正常 Shell。
- 应用关闭时运行中任务有可解释归宿。
- Provider 真实连接、列表、上传、下载、删除的常见失败能被统一表达。
- 发布前有固定验收命令和人工冒烟路径。

## 2. 推荐执行切片

### 2.1 弹窗与消息

- 实现统一错误详情弹窗。
- 实现统一确认弹窗。
- 连接测试成功和失败均可展示详情。
- 启动失败使用最小错误界面，不进入正常 Main Shell。

验收：

- 删除远程对象前必须确认。
- 清理历史前必须确认。
- 重置设置前必须确认。
- 错误详情不包含 secret material。

### 2.2 Application 结果收口

- 连接测试结果包含 provider、目标摘要、错误类别和可读原因。
- 远程路径上下文包含上传、删除、分页、root/bucket 视图等能力标志。
- 传输队列和历史结果包含失败/取消/中断摘要和可重试标志。
- 错误详情快照由 Application 组织，Presentation 只负责展示。

验收：

- UI 按能力标志禁用按钮，不按 provider id 写死规则。
- 可重试状态由 Application 明确给出。
- Application 不返回弹窗指令。

### 2.3 远程浏览闭环

- 加载失败展示摘要、重试入口和详情入口。
- 删除操作接入确认弹窗。
- 上传 / 下载创建任务后有成功或失败反馈。
- 分页按钮状态稳定，不出现无下一页仍可点击的状态。

验收：

- bucket/root 视图禁用不适用操作。
- 失败后可以重试当前加载请求。
- 删除失败可以查看详情。

### 2.4 传输体验闭环

- 队列页展示成功、失败、暂停、中断等状态。
- 历史页展示成功、失败、取消、中断。
- 失败和中断可以查看详情。
- 可重试任务提供重试入口。
- 应用关闭时运行中任务落为 `Interrupted` 或等价可解释状态。

验收：

- 取消不显示为失败。
- 中断不显示为普通失败。
- 重试通过 Application 用例，不直接操作 Transfer Runtime。

### 2.5 启动和恢复诊断

- 配置损坏阻断正常 Shell。
- 凭据存储不可用阻断正常 Shell。
- 传输状态存储不可用阻断正常 Shell。
- 启动失败界面展示脱敏详情、诊断路径、恢复入口和退出入口。

验收：

- Infrastructure 不弹 UI。
- Application 不负责启动错误界面。
- 启动失败详情不泄漏 secret material。

### 2.6 Provider 实用性加固

- 连接测试执行真实轻量后端操作。
- 常见错误映射为 Core 错误类别。
- Provider 错误详情保留脱敏技术信息。
- 每个 Phase 8 provider 有手工验收步骤。

验收：

- 错误凭据、无权限、网络失败、路径不存在等失败能显示统一摘要和详情。
- SDK exception 不穿透到外层。
- 手工验收不要求提交真实密钥或 token。

## 3. 发布前固定检查

每轮 Phase 9 结束前至少执行：

```powershell
dotnet restore AtomBox.sln
dotnet build AtomBox.sln
dotnet test AtomBox.sln
```

桌面冒烟：

- 启动 Desktop。
- 新增或编辑账号。
- 测试连接成功和失败各一次。
- 进入远程浏览页。
- 触发列表成功和失败各一次。
- 创建上传或下载任务。
- 查看队列和历史。
- 触发删除确认但可以取消。

边界检查：

- Core 不依赖外层模块。
- Application 不引用 Avalonia、AtomUI 或具体 SDK。
- Presentation 不直接调用 provider、Transfer Runtime 或 Infrastructure 存储。
- Provider 不把 SDK DTO 或 SDK exception 传到 Core / Application / Presentation。
- 日志、错误详情和诊断信息不包含 secret material。

## 4. 能力矩阵草案

| 能力 | Phase 9 目标状态 |
| --- | --- |
| 账号管理 | 支持新增、编辑、删除前置校验、连接测试详情。 |
| 远程浏览 | 支持列表、分页、加载失败重试、错误详情。 |
| 上传 / 下载 | 支持创建任务、队列展示、失败详情、可重试入口。 |
| 删除远程对象 | 支持确认、执行、成功反馈、失败详情。 |
| 传输历史 | 支持成功、失败、取消、中断展示。 |
| 启动失败 | 支持阻断正常 Shell 和最小诊断界面。 |
| Provider | 已接入 provider 做实用性加固，不新增 provider。 |
| OAuth | 不做完整授权流程。 |
| 断点续传 | 不做。 |
| 自动恢复任务 | 不做，用户显式重试。 |

## 5. Phase 9 结束判定

Phase 9 可以结束的条件：

- `roadmap.md` Phase 9 完成条件全部满足。
- 本文档固定检查均有结果。
- 关键用户路径可在桌面应用中完成冒烟。
- 已知剩余问题不破坏 MVP 可试用闭环，并被记录为 Phase 10 或后续增强。
- 相关文档从“解冻修订中 / 解冻草案”重新冻结。

## 11. Phase 9：产品可试用性收口 / Release Readiness

目标：把 Phase 8 后已经跑通的工程 MVP 收敛成可试用 MVP。Phase 9 不追求新增大能力，而是补齐误操作防护、错误可诊断、启动和关闭可靠性、传输状态可解释、真实 provider 可用性细节和发布验收清单。

阶段定位：

- Phase 8 完成后，AtomBox 已具备工程意义上的最小闭环：账号、远程浏览、真实 provider、上传下载删除、传输队列和历史链路已经贯通。
- Phase 9 的完成标志，是用户可以在本机以有限风险试用 AtomBox，并且失败时能看到原因、能重试或恢复、不会因为常见误操作造成不可解释的数据损失。
- Phase 9 仍属于 MVP 收口，不进入高级版本能力。
- 详细验收清单见 `docs/implementation/phases/phase-09-release-readiness.md`。

推荐顺序：

1. Presentation 弹窗与消息收口：错误详情弹窗、删除确认弹窗、连接测试详情、启动失败提示、操作结果摘要。
2. Application 结果和上下文收口：统一错误展示快照、连接测试结果展示数据、远程路径能力上下文、账号配置校验结果。
3. 远程浏览页产品闭环：删除确认接入、加载失败详情和重试、分页状态一致性、上传下载创建反馈、当前路径能力禁用规则。
4. 传输体验闭环：失败原因展示、失败任务重试、取消和中断状态区分、应用关闭时运行中任务落为可解释状态。
5. Infrastructure 启动和恢复诊断：配置损坏、凭据存储不可用、状态存储不可用时阻断正常 Shell，并提供诊断路径和恢复入口。
6. Provider 实用性加固：连接测试更贴近真实后端、错误映射补齐、provider 默认文案和字段提示收口、真实 provider 手工验收脚本。
7. MVP 发布验收清单：受支持 / 实验性 / 暂不支持能力矩阵、固定验证命令、桌面冒烟步骤、边界扫描和 secret 泄漏检查。

允许做：

- 新增或调整 Desktop / Presentation 弹窗、消息、ViewModel 状态和只读展示 DTO。
- 调整 Application 结果对象，使 UI 能稳定展示错误摘要、错误详情、能力标志和连接测试结果。
- 增加远程浏览、传输队列、传输历史的失败详情、重试、确认、禁用规则和反馈状态。
- 增加 Infrastructure 启动诊断、配置损坏提示、状态存储失败提示和恢复入口。
- 增加 Transfer 关闭阶段的状态落盘规则，例如运行中任务落为 `Interrupted` 或等价的可解释终态。
- 增加 provider 错误映射、连接测试路径和手工验收文档。
- 增加发布前检查清单和固定验证命令。

禁止做：

- 新增 provider 类型。
- 做完整 OAuth 授权流程、后台 token refresh 服务或 provider session pool。
- 做断点续传、复杂并发调度、分片续传恢复、全局限速或任务自动恢复。
- 让 UI 直接调用 provider、Transfer 内部队列或 Infrastructure 存储实现。
- 让 SDK DTO、SDK exception、secret material 或 provider session 穿透到 Core / Application / Presentation。
- 为了快速修 bug 绕开 Application 用例边界。
- 把诊断信息、日志或错误详情变成 secret material 泄漏通道。

完成条件：

- 删除远程对象、清理历史、重置设置等破坏性操作都必须有确认弹窗。
- 所有用户可见失败都至少包含摘要；关键失败可以打开错误详情，详情不得包含 secret material。
- 连接测试成功和失败都有稳定展示，失败时能看到 provider、错误类别、可读原因和必要的下一步提示。
- 远程浏览页加载失败可重试，分页按钮状态稳定，上传 / 下载 / 删除操作有明确反馈。
- 传输队列和历史能区分成功、失败、取消、中断；失败和中断能展示原因；可重试任务提供重试入口。
- 应用关闭时运行中的传输任务不会被伪装成成功或普通失败，必须持久化为可解释状态。
- 配置、凭据、状态存储等启动关键依赖失败时，不进入正常业务 Shell，而是进入诊断 / 恢复提示。
- Provider 错误映射覆盖常见认证失败、网络失败、权限失败、路径不存在、限流或服务端错误。
- 发布验收清单中的 restore / build / test / desktop smoke / boundary scan / secret scan 均可执行并记录结果。

阶段完成后，AtomBox 可以被称为“可试用 MVP”。后续 Phase 10 才进入真实使用打磨；高级传输能力、自动恢复、完整 OAuth 流程或更复杂产品功能仍然延后。
