# AtomBox 专业化路线图

> 文档状态：后续产品能力规划
>
> 创建时间：2026-07-05
>
> 范围：记录 v0.1 后除新增 Provider 外的专业化能力方向、推荐优先级和实施边界。

本文档不是 `roadmap.md` 的替代品。`roadmap.md` 仍然负责工程阶段、发布基线和跳步规则；本文档只记录 AtomBox 从“可用”走向“专业可长期使用”时应优先打磨的能力。

Provider 扩展继续按 `docs/modules/providers/` 和真实测试矩阵推进。本文档重点讨论不依赖新增存储类型也能提升专业度的产品能力。

## 推荐优先级

1. 批量文件管理和专业传输队列。
2. 传输可靠性、冲突策略、失败恢复和错误可解释性。
3. 大文件、校验、断点续传和分片续传恢复。
4. 本地与远端目录对比、单向同步和 dry-run。
5. 诊断日志、请求追踪、诊断包导出和 secret 脱敏。
6. 安装包、自动更新、版本信息和发布说明。

## 传输能力专业化

- 支持任务暂停、继续、重试、取消和优先级调整。
- 支持全局限速、单任务限速和并发数配置。
- 为上传、下载、复制、移动、删除等任务统一冲突策略：覆盖、跳过、重命名、仅较新文件覆盖。
- 补齐大文件路径：分片上传、分片下载、断点续传、失败后续传恢复。
- 对支持校验的 Provider 记录 checksum、ETag 或 provider 原生校验字段。
- 传输队列需要保留可解释状态：等待中、运行中、重试中、已暂停、已失败、已完成。

## 文件管理专业化

- 远端文件列表支持多选、批量删除、批量下载、批量移动、批量重命名。
- 支持文件夹递归上传、递归下载、递归删除。
- 对对象存储的伪目录语义做清晰处理，避免把 OSS bucket 内的 prefix 误当成真实目录。
- 支持排序、筛选、搜索、复制路径、复制对象 URL、生成临时访问链接。
- 文件详情面板展示大小、时间、类型、provider 元数据、权限或公开访问状态。
- 支持拖拽上传和从远端拖拽到本地的下载入口。

## 账号与安全

- 连接测试需要输出结构化结果：认证是否成功、bucket/list 权限、读权限、写权限、删除权限、endpoint 是否匹配。
- 账号导入导出默认不包含明文密钥；如必须导出密钥，应使用本地加密或明确交互确认。
- 日志、诊断包、异常文案和崩溃信息必须脱敏 AK/SK、token、签名 URL 和本地敏感路径。
- 高风险操作需要审计记录：批量删除、覆盖、跨 Provider 复制、权限变更。
- 后续如接入 OAuth 网盘，应先设计 token 生命周期、刷新失败处理和撤销授权路径。

## 同步与对比

- 优先做本地目录与远端目录对比，展示新增、修改、删除、冲突和无法判断的项目。
- 单向同步先于双向同步落地，并且必须提供 dry-run 预览。
- 同步规则需要支持 include / exclude、文件大小、修改时间和扩展名过滤。
- 双向同步、冲突自动合并和后台常驻同步暂不作为第一阶段目标。

## 可观测性与诊断

- 每个传输任务保留关键事件：创建、开始、重试、失败、恢复、完成。
- Provider 异常需要尽量保留原始错误码、request id、endpoint、bucket、object key，但不能泄漏 secret。
- 提供诊断包导出入口，包含应用版本、运行平台、provider 类型、近期日志、任务摘要和脱敏后的配置摘要。
- 统计传输速度、队列等待时间、失败次数和重试次数，帮助定位网络、权限或 provider 限制问题。

## 发布产品化

- Windows 侧补齐安装包、开始菜单快捷方式、卸载入口和自动更新策略。
- macOS 侧补齐签名、公证、dmg 或 zip 发布策略。
- Linux 侧优先考虑 AppImage，再评估 deb / rpm。
- 发布流程应产出 changelog、版本号、构建信息和可复现的 artifact 命名。
- 应用内展示版本信息、构建时间和检查更新入口。

## 当前非目标

- 不把新增 Provider 当作专业化的唯一方向。
- 不在没有 dry-run 的情况下实现双向同步。
- 不在日志、诊断包或错误弹窗中暴露密钥。
- 不绕过 Application / Provider 契约直接在 UI 中实现传输、同步或账号能力。

## 实施前置要求

新增上述能力前，需要同步更新相关模块文档、UI 文档和测试矩阵。涉及真实 Provider 行为的能力，应优先补充 headless 真实性测试，再进入桌面 UI 收口。
