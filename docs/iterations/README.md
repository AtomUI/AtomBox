# Iterations 文档入口

> 文档状态：长期迭代文档规范
>
> 创建时间：2026-07-05
>
> 目的：说明日常版本升级、增强型功能、设计决策和版本验收记录的组织方式。

`docs/iterations/` 用于承载 AtomBox v0.1 之后的日常迭代文档。

本目录记录“某个需求为什么做、做什么、怎么设计、怎么验收、完成后同步了哪些正式设计文档”。它不替代 `architecture/`、`modules/`、`ui/` 和 `implementation/`，也不把历史 Phase 文档继续扩展成日常需求池。

## 阅读顺序

1. 需要了解候选需求池时，读 `backlog.md`。
2. 需要了解某个版本计划时，进入 `releases/` 阅读对应版本文档。
3. 需要实现某个具体增强功能时，进入 `features/` 阅读功能设计文档。
4. 需要追溯长期技术取舍时，进入 `decisions/` 阅读决策记录。

## 目录职责

| 路径 | 职责 |
|---|---|
| `backlog.md` | 记录候选能力池、粗优先级和当前状态，只放简短条目。 |
| `features/` | 记录单个增强需求的背景、目标、非目标、设计方案、测试验收和文档同步要求。 |
| `releases/` | 记录某个版本的迭代范围、验收清单、发布前检查和最终交付状态。 |
| `decisions/` | 记录影响架构、依赖、跨平台能力或长期维护成本的设计决策。 |

## 文档关系

- `docs/iterations/` 记录迭代过程和需求上下文。
- `docs/architecture/` 记录当前有效的架构边界。
- `docs/modules/` 记录当前有效的模块契约和实现约束。
- `docs/ui/` 记录当前有效的桌面交互和页面规则。
- `docs/implementation/testing/` 记录长期可复用的测试矩阵。

功能完成后，稳定事实必须同步到正式文档。若迭代文档与正式文档冲突，以正式文档为准，并应回头补充迭代文档的状态说明。

## Feature 文档模板

```md
# 功能名称

> 状态：Draft / Planned / In Progress / Done / Superseded
> 目标版本：v0.x.x / TBD
> 创建时间：YYYY-MM-DD
> 影响模块：Core / Application / Desktop / Providers / Transfer / Infrastructure

## 背景

说明为什么要做，用户场景是什么。

## 目标

列出本次必须完成的能力。

## 非目标

明确本次不做什么，避免范围膨胀。

## 用户体验

描述入口、交互、状态、错误提示和边界行为。

## 设计方案

说明模块职责、数据流、关键对象和接口变化。

## 测试与验收

列出单元测试、集成测试、手工 smoke test 和验收标准。

## 文档同步

列出完成后需要同步更新的正式文档。
```

## 维护规则

- `backlog.md` 不写大段设计，避免成为无法维护的需求散文。
- 每个可独立实现的增强功能都应有一份 `features/*.md`。
- 每个正式发版版本都应有一份 `releases/vX.Y.Z.md`。
- 涉及跨平台能力、重依赖、外部 SDK、长期兼容策略的取舍，应补充 `decisions/*.md`。
- 完成功能实现前，必须确认 feature 文档、正式设计文档和测试要求已经对齐。
